# Логічні модулі Transport Management System

Документ описує логічні модулі системи управління транспортом (Transport Management System), реалізовані у вигляді трьох мікросервісів.  
Кожен сервіс має власну модель даних, правила предметної області та механізми взаємодії з іншими сервісами.

---

## 1. TechnicalService

**Контекст:** Управління технічним станом та обслуговуванням транспорту  
**Тип БД:** SQL (MySQL)  
**Технології:** ADO.NET + Dapper  

### Призначення
Відповідає за повний життєвий цикл технічного обслуговування автобусів:  
- реєстрація автобусів;  
- відстеження технічного стану;  
- облік технічних оглядів;  
- ведення історії ремонтів і обслуговувань;  
- облік використаних запчастин.

### Основні сутності

| Сутність | Опис |
|----------|------|
| Bus | Одиниця транспорту |
| TechnicalExamination | Проведений технічний огляд |
| BusMaintenanceHistory | Планові та фактичні роботи з обслуговування |
| BusStatus | Актуальний стан автобуса |
| RepairPart | Запчастини для ремонту |
| ExaminationRepairPart | M:N зв’язок між оглядом і запчастинами |

### Межі контексту
TechnicalService керує:  
- технічним станом;  
- ремонтами;  
- оглядами та запчастинами;  
- статусами автобусів (`В роботі`, `На ремонті`, `Несправний`).  

Контекст не містить інформації про маршрути, рейси чи персонал.

---

## 2. RoutingService

**Контекст:** Управління маршрутами та експлуатацією  
**Тип БД:** SQL (MySQL)  
**Технології:** EF Core Code First  

### Призначення
Планування та управління щоденною експлуатацією автобусного транспорту:  
- маршрути;  
- зупинки;  
- розклади;  
- маршрутні листи;  
- рейси;  
- призначення автобусів на маршрути.

### Основні сутності

| Сутність | Опис |
|----------|------|
| Route | Маршрут (номер, назва) |
| RouteStop | Зупинки маршруту |
| RouteStopAssignment | M:N зв'язок Route ↔ Stop |
| RouteSheet | Маршрутний лист для конкретного автобуса |
| Schedule | Графік руху |
| Trip | Окремий рейс |
| BusInfo | Локальний snapshot даних автобуса |

### Межі контексту
RoutingService відповідає за:  
- структуру маршрутів;  
- зупинки та їх порядок;  
- планування рейсів;  
- фактичні поїздки;  
- використання автобусів на маршрутах.  

Цей контекст не керує технічним станом автобусів, лише отримує інформацію про них.

### Чому існує BusInfo?
`BusInfo` — обмежена копія (snapshot) сутності `Bus` з TechnicalService.  

Причини дублювання:  
- зменшення зв’язності сервісів;  
- швидке читання без запитів між мікросервісами;  
- незмінність історичних даних у `RouteSheet` та `Trip`.  

Таким чином, RoutingService не залежить від наявності живого TechnicalService API.

## 3. PersonnelService

**Контекст:** Управління персоналом, документами та медичними оглядами  
**Тип БД:** MongoDB (NoSQL)  

### Призначення
Веде облік працівників транспортного підприємства:  
- особисті дані водіїв;  
- документи (права, медичні довідки, допуски);  
- медичні огляди;  
- журнал робочих змін.

### Основні колекції

| Колекція | Опис |
|----------|------|
| Personnel | Інформація про працівників |
| PersonnelDocument | Додаткові документи працівника |
| PhysicalExamination | Медичні огляди |
| WorkShiftLog | Історія змін та робочих годин |

### Межі контексту
Контекст відповідає за:  
- медичну придатність персоналу;  
- документи;  
- доступ до роботи;  
- історію змін і відпрацьованих годин.  

PersonnelService не має інформації про маршрути, але у майбутньому може передавати дані про доступність водіїв до RoutingService.
